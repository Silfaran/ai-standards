# Internationalization (i18n) Standards

## Philosophy

- Decide before the first feature: a system that ships in one locale today and adds a second one in six months pays the cost of a retrofit, not the cost of i18n. The retrofit is one to two orders of magnitude more expensive.
- Separate **what is translated by humans** (UI strings, marketing copy) from **what is translated by users or generated content** (a professional's bio, a chat message, a product description). They have different storage, different lifecycles, and different fallback rules.
- The locale negotiated for a request is part of the request, not part of a session. The same user with the same account may legitimately receive different locales on different requests (browser language change, deep link with `?lang=…`, A/B test). Carry the locale explicitly.
- Fallback chains are fixed at the project level, declared once, and applied uniformly. Ad-hoc "if French is missing, try English; or maybe Spanish; depends on the page" decays into ghosts of partial translations.
- A missing translation is a defect, not a runtime fallback excuse. The system MAY fall back, but logs the miss with enough context for a human to fix it.

---

## When this standard applies

This standard applies when ANY of these is true:

- The product ships, or plans to ship within 12 months, in more than one language
- The product accepts user-generated content from speakers of more than one language (chats, profiles, reviews)
- The product targets users in more than one country (date/number/currency formatting differs even when the language does not — `1,234.56` vs `1.234,56`)

If none is true, the standard is read-only and the project keeps its strings in source.

## Vocabulary

| Term | Meaning |
|---|---|
| **Locale** | Language + region tag per BCP 47: `es`, `es-ES`, `es-MX`, `en-GB`. The system MUST be precise: `es-MX` and `es-ES` are different locales for currency, date and some plural rules |
| **UI strings** | Text written by the team and shipped with the product: button labels, error messages, headings. Human-translated, version-controlled with the code |
| **Content translations** | Text describing user-owned data: a professional's bio, a course description, a tag name. May be human-translated, machine-translated, or user-supplied per locale |
| **Reference data translations** | Translations of *system catalogs* (taxonomies, enums presented to users, country names). Centrally curated, sometimes drawn from external standards (ISO 639, CLDR) |
| **Fallback chain** | The ordered list the system walks when a translation is missing: `request.locale → fallback_1 → fallback_n → source language` |
| **Source locale** | The locale the strings/content were originally written in. Always present, never falls back. Typically `es` or `en` |

## Locale negotiation (the contract)

### Inputs (priority order)

1. **Explicit URL parameter** — `?lang=fr` or path prefix `/fr/...`. Highest priority because the user just clicked it.
2. **Authenticated user preference** — stored in the user profile, applied when the user is signed in and no URL override is present.
3. **`Accept-Language` header** — RFC 7231 quality values respected. Negotiated against the supported locales using BCP 47 lookup.
4. **Project default locale** — declared in configuration; used when nothing else matches.

The negotiated locale is bound to the request once, in middleware, and exposed as `Locale` value object. Handlers and view layers read it from the request, never re-negotiate.

```php
namespace App\Domain\Localization;

use Webmozart\Assert\Assert;

final readonly class Locale
{
    private function __construct(public string $tag) {}

    public static function from(string $tag): self
    {
        Assert::regex($tag, '/^[a-z]{2}(-[A-Z]{2})?$/', 'Invalid BCP 47 tag');
        return new self($tag);
    }

    public function language(): string
    {
        return explode('-', $this->tag)[0];
    }

    public function region(): ?string
    {
        $parts = explode('-', $this->tag);
        return $parts[1] ?? null;
    }
}
```

### Outputs (mandatory response headers)

Every response that varies by locale MUST set:

```
Content-Language: es-ES
Vary: Accept-Language
```

`Vary: Accept-Language` is non-negotiable for any cacheable response — without it, intermediaries serve the wrong language to the next user. See `caching.md` (CA-005).

When the negotiation falls back, the response uses the locale actually rendered (not the requested one), and includes `Content-Language` so the client knows.

## UI strings (developer-authored)

UI strings live in JSON files under `src/i18n/{locale}/{namespace}.json`. The format is flat key→value, with namespaces by feature area.

```
src/i18n/
  es/
    common.json
    boards.json
    auth.json
  en/
    common.json
    boards.json
    auth.json
```

```json
// src/i18n/es/boards.json
{
  "title": "Mis tableros",
  "create_button": "Crear tablero",
  "delete_confirm": "¿Eliminar el tablero \"{name}\"? Esta acción no se puede deshacer.",
  "items_count": {
    "one": "{count} tarea",
    "other": "{count} tareas"
  }
}
```

Rules:

- **Source locale is the canonical one.** All keys exist in the source-locale file. Other locales MAY have missing keys — the system falls back, but the missing key is logged.
- **Keys are stable identifiers, not English text.** Use `boards.delete_confirm`, never `Are you sure you want to delete?`. When marketing changes the wording, the key stays; the value changes.
- **Plurals via ICU MessageFormat shape** (`one`, `other`, `few`, `many`, `zero`) — the JSON encodes the shape; the rendering library applies CLDR rules per locale. `one` and `other` cover most languages; `few`/`many` are needed for Slavic, Arabic, Welsh.
- **Interpolation uses `{name}` placeholders.** No string concatenation in code (`'Hola ' + name` is forbidden — breaks word order in non-Latin languages).
- **No HTML in JSON values.** When formatting requires emphasis or links, the rendering layer composes them — see "Rich text" below.
- **No business logic in keys.** Keys do not change when business state changes; values do.

### Rich text (links, emphasis inside a sentence)

When a translated sentence contains a link or emphasis, the JSON value uses placeholder slots and the view composes the rich content:

```json
"terms_notice": "Al continuar aceptas las <terms>condiciones</terms> y la <privacy>política de privacidad</privacy>."
```

The renderer maps `<terms>...</terms>` to a `<router-link>` and `<privacy>...</privacy>` to another. Translators see meaningful tags, not HTML — fewer broken layouts and zero XSS risk via translation.

### What never goes in the i18n files

- Translations of user-owned data (a board name, a user bio) — those are **content translations** below.
- Logs, error messages aimed at developers, span attributes, metric labels — operational signals are always in the source language.
- API error codes (`error.invalid_credentials`) — the API returns the code, the frontend looks up the human message.

## Backend response strings

API responses MUST distinguish between machine-readable codes and human-readable messages:

```json
// CORRECT — code is a stable contract; message is for humans, localized
{
  "error": "invalid_credentials",
  "message": "El correo o la contraseña no son válidos.",
  "details": []
}
```

The `error` field never changes per locale. Clients switch on `error`, not on `message`. The `message` is the backend's negotiated-locale rendering.

When the backend serves an API consumed only by SPAs, the project MAY skip server-rendered messages and let the frontend render them from the code — recorded as a project ADR. Mixed approach (some endpoints render, others don't) is forbidden — pick one and document it.

## Content translations (user-owned, taxonomy, multi-locale entities)

These are entities whose textual fields exist in multiple locales: a course title, a microhabilidad name, a professional's bio, a service description, a country name in a dropdown.

### Storage models — choose one per project, recorded as ADR

**Option A — `translations` polymorphic table (recommended for catalogs and curated content)**

```sql
CREATE TABLE translatable_entities (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    -- aggregate-specific columns
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE translations (
    entity_type TEXT NOT NULL,      -- 'course', 'taxonomy_node', 'professional_bio'
    entity_id   UUID NOT NULL,
    field       TEXT NOT NULL,      -- 'title', 'description'
    locale      TEXT NOT NULL,      -- BCP 47
    value       TEXT NOT NULL,
    source      TEXT NOT NULL,      -- 'human', 'machine', 'user'
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (entity_type, entity_id, field, locale)
);

CREATE INDEX idx_translations_entity ON translations (entity_type, entity_id);
```

When to pick: catalogs with many translated fields per row, frequent additions of new locales, translations curated by a separate workflow (CMS, translation vendor).

**Option B — JSONB column on the entity (recommended for sparse, low-cardinality cases)**

```sql
CREATE TABLE professionals (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    -- ...other columns
    bio JSONB NOT NULL DEFAULT '{}'::jsonb
    -- bio = {"es": "...", "en": "...", "fr": "..."}
);

CREATE INDEX idx_professionals_bio_es ON professionals ((bio ->> 'es'));
```

When to pick: one or two translated fields per row, locales bounded and rarely change, no separate translation workflow. JSONB queries are slower and less indexable than normalized tables — don't use this if the project has 5+ translated fields per aggregate.

**Anti-pattern — separate columns per locale**

```sql
-- WRONG — adding a locale is a migration; cannot iterate locales generically
CREATE TABLE products (
    name_es TEXT,
    name_en TEXT,
    name_fr TEXT
);
```

This pattern is forbidden. Adding `de` requires a schema migration AND code changes everywhere. Use Option A or B.

### Source-of-truth field

Every translatable entity declares its **source locale** — the locale the original was written in:

```sql
ALTER TABLE professionals ADD COLUMN source_locale TEXT NOT NULL DEFAULT 'es';
```

The source-locale value is never machine-translated, never expires, and is the final fallback in the chain.

### Query rule

A repository fetching a translatable entity for display takes the locale as an explicit parameter and returns the field value already resolved against the fallback chain — handlers do NOT do `if ($entity->bio['fr'] ?? $entity->bio['en'] ?? ...)`:

```php
public function findByIdLocalized(string $id, Locale $locale): ?ProfessionalView;
```

The repository (or a `LocalizedFieldResolverService` it uses) walks the chain and returns a `ProfessionalView` DTO with strings already in the negotiated locale. The view layer never sees the raw `bio` JSONB.

## Fallback chain

A single fallback chain is declared per project in `{project-docs}/i18n.md` (or in the same file as `decisions.md` if the project keeps everything together). The chain is global — never per-feature — and follows three rules:

1. **The negotiated locale is always first.** If the request is `fr-CA`, the system tries `fr-CA`, then walks the chain.
2. **The chain ends at the source locale.** The source locale always has a value; the system never returns `null` to the view layer because of i18n.
3. **The chain is monotonic by language family when possible.** `fr-CA → fr → en → es` is sensible; `fr-CA → de → es` is not.

Example, for a Spanish-source product targeting LATAM and Europe:

```
es-MX → es → en   (when source is es)
fr-FR → fr → en → es
en-US → en → es
```

A miss in the chain MUST emit a structured log line:

```json
{
  "event": "i18n.missing",
  "namespace": "boards",
  "key": "delete_confirm_with_dependencies",
  "requested_locale": "fr-FR",
  "served_locale": "en",
  "level": "warn"
}
```

Aggregating this log line is how the team finds out which strings need translation, without users having to file a ticket. The `i18n.missing` rate is a quality SLI — see `observability.md` SLO shape.

## Plurals, dates, numbers, currency

The application MUST NOT format these values manually. Locale-aware formatting is delegated to the platform's i18n library:

| Value | Backend (PHP) | Frontend (TS/Vue) |
|---|---|---|
| Plurals | Symfony Translator with ICU format | `vue-i18n` `$tc()` or Intl `PluralRules` |
| Dates | `IntlDateFormatter` | `Intl.DateTimeFormat` |
| Numbers | `NumberFormatter::DECIMAL` | `Intl.NumberFormat` |
| Currency | `NumberFormatter::CURRENCY` (paired with `Money` VO from `payments-and-money.md`) | `Intl.NumberFormat` with `style: 'currency'` |
| Relative time ("hace 2 días") | `IntlDateFormatter` + `RelativeTimeFormatter` | `Intl.RelativeTimeFormat` |

**Forbidden patterns:**

- `date('d/m/Y')` — assumes locale, breaks for `en-US` and Asian formats.
- `number_format($n, 2, '.', ',')` — hardcodes the separators of the source locale.
- `'$' . $price` — currency symbol is locale-dependent (€, $, £, ¥, kr) and so is its position.
- `($count === 1 ? 'item' : 'items')` — only handles English plural rules; broken for Polish, Arabic, Russian.

## User-generated content (chat, comments, profiles)

User-supplied content is stored in the locale the user wrote it in, with the locale recorded:

```sql
CREATE TABLE chat_messages (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    thread_id UUID NOT NULL,
    author_id UUID NOT NULL,
    body TEXT NOT NULL,
    body_locale TEXT NOT NULL,           -- detected or user-declared
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Translation of user content (e.g. a Spanish chat seen by a French recipient) is rendered on demand. The translation pipeline is:

1. User requests the message in their locale.
2. Backend checks a `content_translations` cache (same shape as the `translations` table above, with `source = 'machine'`).
3. On miss, calls the translation provider via `TranslationGatewayInterface` (see `llm-integration.md` for the gateway pattern).
4. Stores the result, marks `source='machine'`, returns the translation.
5. Annotates the rendered text as machine-translated in the UI ("traducido automáticamente"), per UX honesty rules.

Machine translations have a TTL — they MAY be invalidated when the original message is edited, or when the translation provider improves. Human edits to a machine translation promote `source` from `machine` to `human` and remove the TTL.

## Reference data (taxonomies, country lists, currencies)

Project-defined catalogs (e.g. status enums, taxonomies) use Option A storage above.

System-wide reference data (country names, currency names, locale display names) MUST come from CLDR via the platform library — never hardcode lists of countries:

| Data | Backend | Frontend |
|---|---|---|
| Country names | `Symfony\Component\Intl\Countries::getName($code, $locale)` | `Intl.DisplayNames(locale, { type: 'region' })` |
| Currency names | `Symfony\Component\Intl\Currencies::getName($code, $locale)` | `Intl.DisplayNames(locale, { type: 'currency' })` |
| Locale display names | `Symfony\Component\Intl\Locales::getName($code, $locale)` | `Intl.DisplayNames(locale, { type: 'language' })` |

A hardcoded list of countries in the codebase is a defect — CLDR is updated yearly and the platform library tracks it.

## Frontend

Frontend uses `vue-i18n` (`tech-stack.md` may be updated to pin the version when this standard ships).

### Bootstrap

```ts
// main.ts (after createPinia, before VueQueryPlugin)
import { createI18n } from 'vue-i18n'

const i18n = createI18n({
  legacy: false,
  locale: navigator.language,           // initial guess; replaced by negotiated locale on auth
  fallbackLocale: ['en', 'es'],         // matches backend chain
  messages: {},                         // loaded async per locale
  missingWarn: import.meta.env.DEV,
  fallbackWarn: import.meta.env.DEV,
})

app.use(i18n)
```

### Lazy-loaded namespaces

Locale JSON files are loaded on demand per route — never bundle every locale upfront:

```ts
async function loadLocale(locale: string, namespace: string): Promise<void> {
  const messages = await import(`@/i18n/${locale}/${namespace}.json`)
  i18n.global.mergeLocaleMessage(locale, { [namespace]: messages.default })
}
```

### Synchronization with backend

When the user signs in, the frontend reads the user's preferred locale from the auth response and re-applies it via `i18n.global.locale.value = locale`. From that moment, every API request includes `Accept-Language: <locale>` (set on the Axios instance interceptor).

A change of locale in the UI:
1. Updates the in-memory `i18n.global.locale.value`.
2. Calls `PUT /api/v1/me/preferences` to persist the change.
3. Reloads the active queries via `queryClient.invalidateQueries()` so server-rendered messages get re-fetched in the new locale.

### Forbidden in components

- String concatenation for translated content: `'Hola ' + name` → use `t('greeting', { name })`.
- Hardcoded English fallback in templates: `{{ t('foo') || 'Foo' }}` masks missing keys.
- Locale-dependent `toLocaleString` calls without an explicit locale argument: `(123.45).toLocaleString()` reads the browser locale, which may differ from the negotiated one.

## Operational discipline

### Translation source attribution

Every entry in the `translations` table records `source`:

| Source | Meaning | Lifecycle |
|---|---|---|
| `human` | Translated by a person; committed via the translation workflow | Stable; revised on content change |
| `machine` | Generated by an LLM or translation API | TTL'd; replaced when source changes; promoted to `human` if reviewed |
| `user` | Provided by the entity owner (a professional writes their own bio in two locales) | Stable; user owns it |

A query that displays user-facing text MAY filter by `source` — for example, a paid-tier feature only shows `human` translations of a course description. The filter is explicit, not implicit.

### Coverage report

A read-only query against `translations` produces, per `entity_type`, the percent of entities with at least one translation per locale. This report runs on cron and surfaces in operations dashboards. A drop in coverage after an import is a defect (someone added 1000 entities without seeding translations).

### Adding a new locale to a live project

The procedure (in order):

1. Add the locale to the supported list in configuration.
2. Update the fallback chain in `{project-docs}/i18n.md`.
3. Seed the UI string files (`src/i18n/<locale>/*.json`) — initially via human translation OR machine translation marked for review.
4. Run the content translation backfill job for any catalog that should be available immediately. Other content stays as machine-translated on first hit.
5. Enable the locale in the locale-picker UI.
6. Watch the `i18n.missing` log volume for two weeks; close the long tail.

A locale enabled before steps 3–4 is broken in production from day one — auto-reject in review.

## Anti-patterns (auto-reject in review)

- Strings hardcoded in templates or PHP code where the file is meant to be user-facing (`<button>Submit</button>`, `throw new \Exception('User not found')` in a validation handler).
- A translation function called with a variable key (`t($dynamicKey)`). Static keys only — otherwise extraction tools cannot find them.
- A new column suffixed with a locale code (`name_es`, `description_en`) — see anti-pattern in storage section.
- Manual date/number/currency formatting (see "forbidden patterns" above).
- API responses with a `message` field but no `error` code — clients cannot localize independently.
- A `Vary: Accept-Language` header missing from a cacheable response that depends on locale.
- A locale negotiated per layer ("the controller picks one, the view picks another"). One negotiation, in middleware, frozen for the request.
